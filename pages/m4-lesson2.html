Module 4 Lesson 2 Resource Management Best Practices

Right-Sizing Requests, Managing Jobs, Storage Discipline, and Getting Help

Every CPU, memory unit, and wall time minute you request is reserved exclusively for your job. SLURM cannot reassign those resources to anyone else while your job holds them, even if the job never touches them. Over-requesting forces your job to wait for a larger block of idle resources, slows down other users' jobs, and lowers your own future queue priority under fair-share scheduling. The fix is to start with a generous estimate, measure what the job actually used, and tighten the request for next time.

The slides below cover how to measure real job usage, how to read pending job reasons and control running jobs, how to put data on the right file system, and where to go when you need help.

Importance of right-sizing requests

Reserved but unused resources cannot be allocated to other jobs. Over-requesting makes your own job wait longer (larger idle block needed), slows others down, and lowers your fair-share priority for future submissions.

sacct for right-sizing

Run sacct -j <jobid> --format=JobName,Elapsed,MaxRSS,NCPUS,State --units=G after a job finishes to see actual runtime, peak memory, and core count. Use these to tighten --time and --mem for next time.

MaxRSS

Peak resident set size: the maximum physical RAM the job used at any point. The primary metric for setting --mem. Add 10-20% headroom above MaxRSS for the next submission.

seff

Displays CPU and memory efficiency for a completed SLURM job in one concise view. Shows what fraction of requested resources were actually consumed.

Wall time padding rule

Set --time to the job's measured Elapsed time plus 20-30%. This protects against runtime variance without wasting large blocks of reserved time.

Exit code 137

The job was killed by the out-of-memory (OOM) killer. The fix is to increase --mem for the next submission.

More cores does not mean faster

Extra cores only help if your code is written to use them (MPI, OpenMP, or embarrassingly parallel workloads). A serial program runs on exactly one core regardless of how many are requested; the rest sit idle.

GPU request verification

GPUs help only if your software offloads work to them (PyTorch, TensorFlow, GROMACS GPU builds). Check live utilization with nvidia-smi in an interactive session.

squeue --me

Shows only your own jobs and their current state.

squeue -p <partition>

Shows all jobs in a given partition. Useful for gauging how busy a partition is before submitting.

scancel

Cancel a job. scancel <jobid> cancels one job. scancel -u $USER cancels all of your jobs.

scontrol show job <jobid>

Displays full detail on a job, including resource allocation, node assignment, and submission parameters.

scontrol hold / scontrol release

scontrol hold <jobid> prevents a pending job from starting. scontrol release <jobid> lets it proceed. Useful when you need to adjust something before a job runs.

squeue --start

Shows estimated start times for pending jobs based on the current queue state.

Pending reason: Priority

Other jobs are ranked higher in the queue. The job will start once those ahead of it finish.

Pending reason: Resources

Not enough free nodes, CPUs, or memory are available right now to satisfy the request.

Pending reason: QOSMaxCPUPerUser

The user has hit the per-user CPU limit for the partition's QOS policy. Clears automatically when one of your other jobs finishes and releases cores.

/N/scratch purge policy

Files not accessed in more than 30 days are deleted automatically without notice. Slate Scratch is fast working storage for active jobs, not long-term storage.

Python/conda environments on Slate

pip, conda, and R install packages into your Geode home directory by default. ML environments exhaust both your 100 GB size quota and your 800,000-file inode limit. Redirect installs to Slate: python -m venv /N/slate/$USER/envs/myproject.

IU HPC Guide

The primary reference for commands, workflows, system specs, and best practices. The first place to look for any HPC question.

Submit a ticket

For system errors, account issues, or anything blocking your work. File through RT Projects at https://projects.rt.iu.edu/help/?queue=hps for HPC and Quartz issues.

Well-formed support ticket

Include the SLURM job ID ($SLURM_JOB_ID), a relevant excerpt of your output or error log, which system is affected, and what you already tried. A complete ticket resolves faster.

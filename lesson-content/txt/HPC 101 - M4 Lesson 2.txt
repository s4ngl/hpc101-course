Request only what you need
‚óè Every CPU, memory, and walltime you request is reserved
  exclusively for your job
   ‚óã SLURM cannot give those resources to anyone else while your
     job holds them
   ‚óã this is true even if your job never touches them
Request only what you need
‚óè Over-requesting has real costs:
   ‚óã your job waits longer, because the scheduler must find a larger
     block of idle resources
   ‚óã other users' jobs wait longer, because resources sit reserved but
     unused
   ‚óã IU's HPC systems use fair-share scheduling, so over-requesting
     lowers your future queue priority

‚óè Ask for what the job needs, not the maximum available
Measure actual usage ‚Äî `sacct` and `seff`
 ‚óè Start generous, measure what the job actually used, then tighten
 ‚óè After a job finishes, read its real usage with `sacct`:

[citrain@h2 ~]$ sacct -j 9472 --format=JobName,Elapsed,MaxRSS,NCPUS,State --units=G
   JobName Elapsed MaxRSS                    NCPUS  State
  -------- -------- -------- -------- --------
   align.sh 01:42:18        6.20G          8 COMPLETED

 ‚óè `seff <jobid>` summarizes CPU and memory efficiency in one view
 ‚óè Tighten for next time:
       ‚óã set `--time` from `Elapsed` plus 20‚Äì30%
       ‚óã set `--mem` from `MaxRSS` (peak memory) plus 10‚Äì20%
 ‚óè Exit code `137` = killed by the out-of-memory killer; raise `--mem`
More cores ‚âÝ faster
‚óè More cores help only if your code is written to use them:
    ‚óã MPI (Message Passing Interface), OpenMP, or embarrassingly
      parallel workloads benefit
    ‚óã a serial program runs on exactly one core no matter how many
      you request; the rest sit idle
    ‚óã confirm parallelism at runtime with `top -u $USER`, or after the
      job with `seff`
Not all software uses GPUs
‚óè GPUs help only if your code offloads work to them:
   ‚óã PyTorch, TensorFlow, GROMACS (GPU builds) use the GPU;
     most plain Python, R, and bioinformatics tools do not
   ‚óã in an interactive GPU session, check live activity with `nvidia-smi`
   ‚óã confirm a GPU was allocated with:
          $ sacct -j <jobid> --format=JobID,AllocTRES%40,State

‚óè Don't request hardware your code cannot use

Job control commands recap
‚óè Watch the queue:
        $ squeue --me ‚Äî your jobs only
        $ squeue -p <partition> ‚Äî all jobs in a partition (gauge how busy it is)

‚óè Cancel:
        $ scancel <jobid> ‚Äî cancel one job (also `scancel -u $USER` for all of yours)

‚óè Inspect and control with `scontrol`:
    ‚óã   $ scontrol show job <jobid> ‚Äî full detail on a job
    ‚óã   $ scontrol hold <jobid> / $ scontrol release <jobid> ‚Äî keep a pending job
        from starting, then let it
    ‚óã   $ scontrol suspend <jobid> / $ scontrol resume <jobid> ‚Äî pause and
        resume a running job
Why is my job pending?
 ‚óè A pending job shows state `PD`. the reason is in the
   `NODELIST(REASON)` column of `squeue`:
      ‚óã   `Priority` ‚Äî other jobs ranked higher; yours will start eventually
      ‚óã   `Resources` ‚Äî not enough free nodes, CPUs, or memory right now
      ‚óã   `Dependency` ‚Äî waiting on another job to finish
      ‚óã   `QOSMaxCPUPerUser` ‚Äî you've hit a per-user CPU limit
      ‚óã   `ReqNodeNotAvail` ‚Äî a requested node is unavailable or down

 ‚óè Estimate when it will start: $ squeue --start -j 9472

JOBID PARTITION NAME USER ST             START_TIME NODES
NODELIST(REASON)
9472 gpu train_nn citrain PD 2026-06-13T21:15:42  1 (Priority)

Recap
‚óè `/N/scratch` is purged: files untouched for more than 30 days are deleted
  without notice
    ‚óã it is fast working space, not long-term storage
‚óè Archive to the Scholarly Data Archive (SDA) at project milestones
‚óè Bundle many small files (for example with `tar`) before storing or archiving
‚óè Clean up when you go idle: move results off scratch, delete intermediates
  you no longer need
Where to put what
Slate Scratch                  Active job input/output, intermediate results, temporary working
                               data

Geode Home Directory           Scripts, configs, small source code; things you want to keep but
                               that stay small

Slate Project                  Shared datasets that a research group needs across multiple jobs
                               (request via RT Projects)

Slate                          Scholarly data for data-intensive workflows and analytics running
                               Quartz or Big Red 200.

Scholarly Data Archive (SDA)   Completed datasets and results you want to archive long-term
Package installs fill up Geode fast
‚óè Python (pip), conda, and R all install packages into your Geode home
  directory by default. A single machine-learning environment with PyTorch,
  TensorFlow, or JAX can consume 10‚Äì20 GB. A few environments and your
  100 GB quota and/or your 800,000 file limit could be completely used up.

‚óè Redirect installs to Slate to keep your Geode home directory lean:
   ‚óã For pip (virtualenv):
      # Create the environment on Slate instead of home
      python -m venv /N/slate/$USER/envs/myproject
      source /N/slate/$USER/envs/myproject/bin/activate
      pip install -r requirements.txt

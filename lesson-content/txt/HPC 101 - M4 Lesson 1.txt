Decision table
Scenario                                              Tool

Large files (over 1 GB) between IU systems            Globus

Personal computer to or from an IU system, any size   Globus

Google Drive or OneDrive to or from an IU system      Globus

Quick one-shot copy between two Linux hosts           SCP

Interactive file browsing on a remote system          SFTP

Interactive shell access to the SDA                   HSI

Bundling many small files into SDA archives           HTAR
Constraints
Two constraints override everything in the table:

ā—¸ Protected health information (PHI), any direction: Globus or
  SFTP only
     ā—‹ HSI and HTAR do not encrypt in transit and must never be
       used for PHI

ā—¸ Off-campus to the SDA without an exemption: Globus or SFTP
     ā—‹ HSI and HTAR are on-campus tools; off-campus requires an
       exemption from store-admin@iu.edu

SCP ā€” one-shot copy
ā—¸   Copies a file in a single command, no interactive session
ā—¸   Best for quick, small copies when you know the exact paths
ā—¸   Remote paths take the form `username@hostname:path`
ā—¸   Upload to Quartz:
     ā—‹ $ scp mydata.tar.gz username@quartz.uits.iu.edu:/N/slate/username/
ā—¸ Download from Quartz:
     ā—‹ $ scp username@quartz.uits.iu.edu:/N/slate/username/results.csv .
ā—¸ `-r` copies a directory recursively
ā—¸ Two-Step Login (Duo) required; SSH public key authentication
  available via the SSH public key authentication agreement
ā—¸ No resume: a failed transfer restarts from the beginning
SFTP ā€” interactive browse
ā—¸ Interactive session: browse the remote tree, then transfer
ā—¸ Best for small transfers when you want to navigate before copying
ā—¸ Start a session:
    ā—‹ $ sftp username@quartz.uits.iu.edu
ā—¸ In-session commands: `ls`, `cd`, `pwd`, `put localfile`, `get
  remotefile`, `rename old new`, `exit`
ā—¸ `lls`, `lcd`, `lpwd` act on your local side
ā—¸ Connect to the SDA: $ sftp username@sftp.sdarchive.iu.edu
ā—¸ GUI clients exist: WinSCP (Windows), Cyberduck (macOS)
ā—¸ Two-Step Login (Duo) required; SFTP resumes interrupted transfers

HSI ā€” interactive shell to the SDA
ā—¸ HSI opens an interactive shell on the SDA

ā—¸ Available on all IU research supercomputers; load the module first:
    ā—‹ $ module load hpss

ā—¸ Start a session with `hsi`, then use in-session commands:
    ā—‹ `ls`, `cd`, `pwd`, `mkdir`, `put localfile`, `get remotefile`, `mv old new`,
      `exit`

ā—¸ From Research Desktop (RED):
    ā—‹ Applications > Storage > Scholarly Data Archive
HSI ā€” interactive shell to the SDA
ā—¸ On-campus only; off-campus requires an exemption from
  store-admin@iu.edu

ā—¸ Not PHI-safe: HSI does not encrypt in transit, use Globus or
  SFTP for PHI

ā—¸ SDA deletions are permanent and irreversible
HTAR ā€” bundle many files to tape
ā—¸ HTAR bundles many local files into a single `.tar` archive
  directly on the SDA
    ā—‹ no intermediate local archive needed
ā—¸ The right tool when you have many small files, which the SDA
  handles poorly one by one
ā—¸ Load the module first: $ module load hpss
ā—¸ Create an archive from the current directory:
    ā—‹ $ htar -c -f my_archive.tar *
ā—¸ List contents: $ htar -tf my_archive.tar; extract: $ htar -x -f
  my_archive.tar
HTAR ā€” bundle many files to tape
ā—¸ Limits (more restrictive than the SDA itself):
    ā—‹ max individual file size 68 GB; up to 1 million files per archive
    ā—‹ max directory path 154 characters; max file name 99
      characters

ā—¸ On-campus only; not PHI-safe (use Globus or SFTP for off-
  campus or PHI)
SDA prep rules
ā—¸ Tape is unforgiving of many tiny files: bundle files under 1 MB into
  `.tar` or `.zip` first

ā—¸ Use ASCII-only characters in paths and file names (0x20 to 0x7e);
  rename anything with Unicode first

ā—¸ Archive at milestones: before submitting a paper, after a major
  analysis, at phase changes
SDA prep rules
ā—¸ Date your archive file names so you can keep updated versions and
  find the latest

ā—¸ Keep a separate contents log: the SDA can't preview a `.tar` without
  retrieving it

ā—¸ Delete redundant or superseded data from active file systems to stay
  within quota

Archiving files to the SDA
1. On Quartz, create a multi-gigabyte test file in your Slate directory:
      $ cd /N/slate/username
      $ dd if=/dev/urandom of=testfile.dat bs=1M count=2048
1. Record a checksum so you can compare later:
      $ sha256sum testfile.dat
1. Send the file to the SDA over SFTP:
      $ sftp username@sftp.sdarchive.iu.edu
      in the session: `put testfile.dat`
1. Rename the file on the SDA, still in the SFTP session, then `exit`:
      rename testfile.dat testfile_2026-06-12.dat
Archiving files to the SDA
5.   Pull the renamed file back with Globus (https://globus.iu.edu/):
     source `IURT - Scholarly Data Archive`, destination `IURT - Slate`
     at `/N/slate/username/`
     select `testfile_2026-06-12.dat`, then Start
6.   Confirm the round trip matched by checksum:
     $ sha256sum testfile_2026-06-12.dat
     compare against the value from step 2
7.   Clean up the temporary files:
     on Quartz: $ rm testfile.dat testfile_2026-06-12.dat
     in HSI or Globus: delete `testfile_2026-06-12.dat` from the SDA
